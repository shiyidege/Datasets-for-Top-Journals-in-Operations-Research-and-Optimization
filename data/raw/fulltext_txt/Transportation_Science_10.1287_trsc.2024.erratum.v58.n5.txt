Abstract. The aim of this erratum is to correct an error in the computer implementation of 
the algorithm proposed by Dabia, Ropke, and van Woensel [Dabia S, Ropke S, van Woen­
sel T (2013) Branch and price for the time-dependent vehicle routing problem with time 
windows. Transportation Sci. 47(3):295–454]. Section 6, “Computational Results,” from the 
original paper is rewritten to reflect the corrected implementation, the new computational 
setup, and the updated results.
Keywords:
vehicle routing problem • column generation • time-dependent travel times • branch and price
We became aware that our implementation of the 
branch-and-price algorithm described in the paper by 
Dabia, Ropke, and van Woensel (2013) contains errors. 
This became clear after other researchers contacted us, 
stating that they could not reproduce the results 
reported in the paper. When investigating the program, 
we found an error in the way the travel time functions 
were generated: When generating travel time break­
points, some breakpoints are missing, and as a result, 
travel times can be underestimated.
We suspected that other errors could still be present 
in the old code, and to avoid such mistakes, we imple­
mented the algorithm from scratch. We followed the 
algorithm described in the original paper and did not 
add any extra features. It is also important to stress that 
we have found no error in the algorithms described in 
the article, only in the computer implementation.
Running the new program on the original instances 
showed that results in general improved, and we are 
now able to solve more instances to optimality. Because 
the travel time functions are no longer underestimated, 
the time windows are more constraining, and the 
instances are, as a result, in general, easier to solve. Below, 
we rewrote Section 6, “Computational Results,” from the 
original paper to reflect the corrected implementation, 
the new computational setup, and the updated results.
6. Computational Results
The branch-and-price algorithm is implemented in 
Java on an Intel(R) Core(TM) 2.8-GHz CPU and 16 G 
RAM. For all experiments, we use a time limit of one 
hour. The LP solver Gurobi 8.1.1 is used to solve the lin­
ear programming relaxation of the master problem. For 
our numerical study, we use the well-known Solo­
mon’s data sets (Solomon 1987) that follow a naming 
convention of DTm.n: D is the geographic distribution 
of the customers that can be R (random), C (clustered), 
or RC (randomly clustered); T is the instance type that 
can be either one (instances with tight time windows) 
or two (instances with wide time windows); m denotes 
the number of the instance; and n the number of custo­
mers that need to be served. Road congestion is taken 
into account by assuming that vehicles travel through 
the network using different speed profiles. We consider 
speed profiles that comply with data from real life with 
two congested periods. Speeds in the rest of the plan­
ning horizon (i.e., the depot’s time window) are rela­
tively high. Furthermore, we assume three types of 
links: fast, normal, and slow. Slow links represent links 
within the city center, fast links represent highways, 
and normal links represent the transition from high­
ways to city centers. Moreover, without loss of general­
ity, we assume that breakpoints are the same for all 
speed profiles as congestion tends to happen around 
the same time regardless of the link’s type (e.g., rush 
hours). The link type is chosen randomly and remains 
the same for all instances. The following speed profiles 
are considered.

6.1. TDESPPRC vs. TDSPPRC
In Tables 2–4, we report the instances for which we 
could at least solve the root node of the branch-and- 
bound tree. The first column indicates the name of the 
instance. The columns denoted as “Root LB” show the 
lower bounds obtained in the root node. Moreover, 
the columns denoted as “Best LB” and “UB” indicate, 
respectively, the best lower and upper bounds found 
all over a branching tree. In the column “Time” we 
report the time (in seconds) spent to solve an instance, 
and in the column “Tree” we report the size of the 
branching trees.
For instances with 25 customers, we solved almost all 
instances to proven optimality except for some type 2 
instances. We note that most solved type 2 instances are 
mainly solved when the pricing problem is TDESPPRC. 
In case of instances with 50 customers, most of type 1 
instances and some type 2 instances are solved to opti­
mality. Again the solved type 2 instances are mainly 
solved when the pricing problem is TDESPPRC. For 
instances with 100 customers, only a few type 1 instances 
are solved to optimality, and for some other instances, a 
lower bound is found.
We report results for both TDESPPRC and TDSPPRC 
as the pricing problem. We observe that overall 
TDESPPRC performs better than TDSPPRC except for 
some instances with tight time windows such as C103.50 
and R102.100. Moreover, the lower bounds obtained in 
the root node are worse when using TDSPPRC, which 
explains the larger search trees compared with the case 
of TDESPPRC. In general, as illustrated in the column

6.2. Bidirectional vs. Monodirectional
In Table 6, we illustrate the gains of using a bidirec­
tional search (columns 2–4) over the monodirectional 
search (columns 5–7). We report the best found lower 
and upper bounds in the columns “Best LB” and “UB,” 
respectively. Moreover, in the column “Time” we re­
port the time (in seconds) needed to solve an instance. 
Clearly, the performance of the bidirectional labeling 
algorithm is far better than that of the monodirectional 
version. It can solve more instances and reduces the 
running time with a factor up to 24 times compared 
with the monodirectional version.
