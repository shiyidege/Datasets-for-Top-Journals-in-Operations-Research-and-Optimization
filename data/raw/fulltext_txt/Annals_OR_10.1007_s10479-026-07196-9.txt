Abstract
The resource-constrained project scheduling problem (RCPSP) is a complex optimization
problem aiming to construct feasible schedules that minimize the project makespan while
satisfying the precedence and renewable resource constraints. Priority rule heuristics are
prevalent approaches for solving the RCPSP, particularly in practical applications. However,
these rules are problem-speciﬁc, and no rule can consistently outperform others across differ-
ent projects. Designing priority rules through manual methods requires substantial expertise,
time, and computational effort. This has led researchers to propose automated techniques for
this purpose. Most existing research in this area focuses on unsupervised learning techniques
like genetic programming hyper-heuristics (GPHH), while the investigation of supervised
learning algorithms remains limited. To address this gap, this research explores the potential
of supervised learning algorithms, speciﬁcally regression-based methods, for the automated
designofnewpriorityruleheuristicsforRCPSP.Ninewidelyusedregressionalgorithmswere
evaluated, and the top-performing three were further enhanced using ensemble techniques to
augment their effectiveness. Computational experiments show that regression-based heuris-

1 Introduction
The resource-constrained project scheduling problem (RCPSP) is a common and versatile
challenge encountered in various real-life scenarios such as construction, healthcare, and
software development. It involves constructing a schedule for a set of activities while simul-
taneously considering both the precedence and resource constraints of the project. Given
its relevance in multiple project sectors, effectively solving this problem is crucial. The
primary objective of the RCPSP is to optimize a speciﬁc objective function, such as the
project completion time or cost, while adhering to the aforementioned constraints. Due to
its complexity (NP-hard (Blazewicz et al., 1983)), extensive research has been dedicated to
developing algorithms that can generate feasible solutions for this problem. These algorithms
encompass constructive heuristics, exempliﬁed by the priority rules approach (PR), as well
as advanced (meta-)heuristic approaches like genetic algorithms, and even exact algorithms
such as branch-and-bound procedure and various mixed-integer programming (MIP) formu-
lations. In this research, we explore the use of priority rules in conjunction with regression
algorithms.
Although the PR approach almost always produces inferior solutions compared to meta-
heuristic methods and exact algorithms, its application has never disappeared from the
literature due to its undeniable advantages, such as ease of implementation and the ability to
provide feasible solutions quickly. However, no single rule guarantees superior performance
across all projects or datasets, and developing new (and also good) rules requires substantial
expertise, time and computational effort. Consequently, researchers have developed various
methods to automate the design of priority rules. The objective is to generate a priority rule
that, consistently outperforms not only traditional single rules but also other approaches,
particularly for solving large-scale projects.
Since the PR approach is considered a heuristic approach, one potential avenue for
designing priority rules is to employ hyper-heuristic algorithms. Generally, these algorithms
automate the design of heuristic methods to address complex computational search problems
(Burke et al., 2019). Examples of this can be found in studies like Zhou and J.j. Yang, and Z.
Huang. (2020) and Yu et al. (2017). Hyper-heuristics generate new heuristics by learning from
a set of training instances, with this learning process being either unsupervised or supervised.
In recent years, genetic programming hyper-heuristics (GPHH) has garnered considerable
attention for its role in automating the design of heuristics for the RCPSP and its extensions.
GPHH applies an unsupervised machine learning algorithm that evaluates candidate heuris-
tics on a set of problem instances (i.e., training instances), measures their performance, and
uses the feedback obtained to guide the search toward promising areas in the search space.
Although supervised hyper-heuristics have been extensively studied for problems related to
the RCPSP, such as the job shop scheduling problem, and have shown promising results,

relatively little research has focused on applying supervised learning methods speciﬁcally to
the design of heuristics for the RCPSP. Therefore, our research aims to address this gap by
thoroughly investigating the performance of employing supervised learning approaches to
design new heuristics for the RCPSP.
Supervised learning is an approach within machine learning where machines are trained
using well-labeled data to make accurate output predictions. This labeled data refers to
input data pre-tagged with the correct output. The fundamental objective of a supervised
learning algorithm is to discover a function that correlates the input (independent) variables
(⃗x) with the output (dependent) variable (Y). Supervised learning is generally partitioned
into two distinct types of problems. Classiﬁcation problems are utilized when the output
variable is categorical, representing distinct classes such as "Yes-No" or "High-Medium-
Low". Conversely, regression problems are employed when there is a need to predict a
continuous output variable, often revealing relationships between the input and output, as
seen in weather forecasting or ﬁnancial trend analysis. The use of supervised learning for
heuristic design is a regression task, as it seeks to predict a continuous output value (the
priority value in this research) based on input data and an associated set of features. Our
research focuses on the application of regression algorithms to design heuristics for the
RCPSP.
This research is structured as follows: Sect. 2 provides a succinct overview of pertinent
studies in the ﬁeld. The general methodology of employing regression approaches for the
design of heuristics for the RCPSP, along with a brief introduction to each considered regres-
sion approach, is detailed in Sect. 3. Numerical experiment results and our corresponding
analysis are exhibited in Sect. 4. We conclude and outline future research avenues in Sect. 5.
2 Related work and literature overview
This section provides an overview of the examined problem, substantiated by references
from the existing literature. A concise description of the RCPSP is available in Sect. 2.1.
Subsequently, we introduce the priority rules heuristic in Sect. 2.2. Finally, Sect. 2.3 offers a
comprehensive review of current literature focusing on the automated design of priority rules
for the RCPSP and discusses the application of supervised learning approaches in designing
heuristics for various scheduling problems. It also outlines the motivation for this research.
2.1 Problem statement
The aim of the RCPSP is to schedule a project that satisﬁes the resource and precedence
constraints. In a project, a set of activities needs to be processed without preemption with
a set R of renewable resources (Słowinski, 1981). Set A = {0, 1, . . . , n, n + 1} represents
the set of activities constituting the projects, where activities 0 and n + 1 represent the start
and the end of the project. These two activities are usually referred to as dummy activities
with both processing time (i.e., duration) and resource demand equal to zero, while the other
activities are non-dummy activities with a ﬁxed non-zero duration and a non-zero resource
demand. The resource set R consists of the resources needed for scheduling the non-dummy
activities. Each renewable resource k ∈R has a constant availability of ak per period.
Moreover, activities are linked by their precedence relations. In this research, we consider

2.2 Priority rules heuristic
Because of the complexity of the RCPSP, the problem attracted the attention of many
researchers. Over the last few decades, studies on the scheduling for RCPSP mainly focus
on the exact methods and meta-heuristics that can provide good results by deeply exploring
the solution space. The exact methods search the entire solution space and guarantee opti-
mality when enough CPU time is available, while the meta-heuristics start with an initial
solution and subsequently improve it using principles borrowed from nature through genera-
tions. Summaries of exact methods can be found in overview papers such as Herroelen et al.
(1998) and Weglarz (1999). Recently, Coelho and Vanhoucke (2018) proposed an integrated
branch-and-bound procedure incorporating the best performing components from literature.
Although it shows promising results, projects with a large number of activities may still not
be solved easily, even with today’s powerful computers. Moreover, interesting surveys about
the meta-heuristics are given in Kolisch and Hartmann (2006) and Pellerin et al. (2020).
However, both the exact and meta-heuristic methods are hard to deploy for project managers,
and their runtime is often unacceptable for solving large-sized realistic projects. Therefore,
priority rule-based heuristics have been investigated in the literature since the very beginning
of the search for solutions to the RCPSP. Although their solution quality cannot compete
with the exact methods and meta-heuristics, they are still among the most commonly used
algorithms in practice (-Dumi´c and Jakobovi´c, 2021).
Priority rule-based heuristics consist of two components: a priority rule that assigns a
priority value to each activity and produces an ordered sequence of activities, and a schedule
generation scheme (SGS) that constructs a feasible project schedule based on this sequence.
Both components are brieﬂy reviewed below, and the corresponding pseudocode is provided
in Appendix A.
Priority rule: A priority rule utilizes activity attributes, each with unique values across
activities, as project information to construct a sequence of activities that satisﬁes net-
work precedence constraints. This sequence, often referred to as a priority list (e.g.,
Lambrechts et al. (2008)) or activity list (e.g., Alcaraz and Maroto (2001)), is determined
123

Annals of Operations Research
by a priority function that employs these attributes to assign scheduling priorities. In this
study, we adopt the term priority list to highlight that it results from the application of
priority rules. The activity attributes can be classiﬁed into the following four categories
(Alvarez-Valdés and Tamarit (1989)), i.e., activity information, scheduling information, net-
work information, and resource information. The activity attributes considered in this
research, derived from 23 widely used priority rules, are outlined in Table 1.
Schedule generation scheme: A schedule generation scheme uses the priority list gen-
erate by the priority rule and aims to generate a feasible schedule by extending the partial
schedule (i.e., a schedule where only a subset of the activities has been assigned a starting
and ﬁnishing time) in a stage-wise fashion. At the start of the schedule generation process,
the partial schedule is empty, and the ﬁrst activity on the list is available to be scheduled.
Subsequently, activities are selected according to their order in the priority list and are put
in the schedule following the rules of the generation scheme. There are two well-known
generation schemes, the serial schedule generation scheme (SSGS) and the parallel sched-
ule generation scheme (PSGS). The SSGS selects the activity one by one from the list and
schedules it as-soon-as-possible in the schedule without violating precedence relations and
renewable resource constraints. In contrast, the PSGS selects the activities available to be
scheduled at each predeﬁned time period and schedules them in the schedule when enough
resources are available. A comprehensive description and theoretical analysis of these two
schedule generation schemes are provided in Kolisch (1996b), while their typical usage and
characterization are discussed in Kolisch (2015).
2.3 Regression-based heuristics in scheduling and research motivation
Scheduling involves a diverse range of problems, each with distinct characteristics. Produc-
tion scheduling problems, such as job shop and ﬂow shop scheduling problems, are closely
related to project scheduling, as they share similar constraints and optimization objectives.
This section ﬁrst reviews regression-based heuristics in production scheduling before moving
on to project scheduling. The discussion then shifts to the motivation for further exploring
regression-based heuristics in project scheduling.
Regression is a widely used supervised learning technique for modeling the relation-
ship between input variables and a continuous target variable. It originated in the ﬁeld of
statistics as a fundamental tool for analyzing relationships between variables, and has since
become a core component in machine learning and data science due to its interpretabil-
ity and ﬂexibility (Bishop and Nasrabadi, 2006). A detailed theoretical introduction, along
with examples of applications, can be found in James et al. (2013), Hastie et al. (2009),
and Montgomery et al. (2021). The application of the regression-based heuristic method-
ology is not unprecedented. It has been previously explored in production scheduling,
particularly in job shop and ﬂow shop scheduling problems and their extensions. An
exhaustive literature review on utilizing regression algorithms (and other methods) for
designing heuristics in job shop scheduling problems can be found in Branke et al. (2015).
Among these studies, several stand out for their innovative approaches and contribu-
tions to the ﬁeld. For instance, El-Bouri et al. (2000) introduced heuristics based on
neural network regression to learn from optimal schedules, an approach later reﬁned by
Gupta et al. (2020). Similarly, Weckman et al. (2008) employed solutions obtained through
an evolutionary algorithm as a learning base for their neural network regression. Further,
Ingimundardottir and Runarsson (2011) utilized a similar process but chose logistic regres-
sion as their analytical tool. Another noteworthy study by Koonce and Tsai (2000) applied

Based on the knowledge and practices established in these areas, the fundamental concept
behind using regression algorithms to construct feasible project schedules is similar to that
in job shop and ﬂow shop problems. Speciﬁcally, it is to make use of existing schedules
that are feasible and preferably optimal, and that provide known start times for each activity
within a set of training projects. Regression methods learn from this training data to establish a
relationship between activity features and activity start times. This trained relation effectively
serves as a newly constructed priority rule, which can then be used to generate feasible and,
ideally, high-quality schedules for new, unseen projects.
Moreover, supervised learning methods are increasingly being recognized for their con-
tributions to the ﬁeld of project scheduling. A noteworthy study in this context is the
research conducted by Guo et al. (2021). The authors employed decision tree algorithms,
known for their high accuracy, to identify the most suitable priority rules for speciﬁc
projects. This methodology demonstrated effectiveness by enabling the selection of the
most appropriate rule for a given project, based on project indicators. More recently,
Teichteil-Königsbuch et al. (2023) used a graph neural network to predict activity start times,
which are not necessarily feasible, and then constructed a feasible schedule by sorting activi-
ties by the predicted start times and applying SSGS. Their experiments showed good solution
quality. Given the encouraging initial results from this line of research, it is evident that super-
vised learning methods offer a compelling avenue for further exploration and development
in project scheduling.
Although supervised learning methods have shown promise, the current landscape of
automated priority rule design for the RCPSP is still largely shaped by unsupervised learning
approaches, more speciﬁcally, the GPHH. While genetic programming (GP) is a versatile
evolutionary computation technique that is also applicable to regression tasks such as sym-
bolic regression (which will be employed in this research), its primary role in the project
scheduling literature has been as the foundation of GPHH, an unsupervised learning frame-
work. Rather than learning a direct mapping between activity features and optimal start
times, as supervised learning approaches like regression-based heuristics do, GPHH evolves
priority rules directly. It achieves this by applying candidate priority rules to solve training
instances and evaluating their performance based on the resulting solution quality, typically
using makespan or other performance metrics as feedback.
The ﬁrst application of GPHH to the RCPSP can be found in Frankola et al. (2008), and
thisapproachhasbeenfurtherdevelopedbysubsequentstudies,including -Dumi´c et al. (2018),
Chand et al. (2018), and Luo et al. (2022). Their experiments demonstrate that GP-designed
rules consistently outperform the best traditional priority rules across all widely used datasets,
particularly for project instances with more than 100 activities.
Despite its success, GPHH faces two signiﬁcant challenges in its application to RCPSP.
First, the performance of output heuristics is generally inferior to meta-heuristic methods
(a common limitation across heuristic algorithms). Second, GPHH demands substantial
computational resources during the training process. To improve upon these challenges,
researchers have incorporated strategies like rollout justiﬁcation (Chand et al., 2019; -Dumi´c
and Jakobovi´c, 2022) and ensemble methods (-Dumi´c and Jakobovi´c, 2021), which have
proven effective in improving the performance of the rules designed by genetic programming.
Additionally, Luo et al. (2023) have presented surrogate models as a solution to alleviate the
computational burden of the training phase in GPHH, effectively reducing the time consump-
tion in the process.

Continuing from our exploration of the predominant role of GPHH in the RCPSP, we argue
that it is crucial to explore the less-traveled path of supervised learning methods, speciﬁcally
regression-based heuristics, as an alternative approach. The merits of this strategy rest in
multiple dimensions. Primarily, this approach can absorb insights from historical, optimal,
or nearly optimal schedules and subsequently generate priority rules that proﬁciently solve
new problem instances. It is important to note that while optimal solutions for projects may
not be unique, the insights derived from them are still invaluable for designing new heuris-
tics. Secondly, the successful track record of regression-based heuristics deriving priority
functions from high-quality schedules makes them an attractive frontier for future research.
In addition, the ceaseless progress in machine learning methodologies and computational
capabilities suggests that exploring supervised learning methods for designing priority rules
for RCPSP may yield notable enhancements to existing research. Finally, regression methods
require signiﬁcantly less training time (several minutes) compared to GPHH (hours or even
days, see Luo et al. (2022, 2023)), making them a more practical and efﬁcient choice for
generating new priority rules.
3 Regression-based heuristics for RCPSP
This section presents the methodology used in this research, which involves utilizing regres-
sion methods to develop heuristics for solving the RCPSP. Sect. 3.1 ﬁrst outlines the general
procedures. Subsequently, Sect. 3.2 and Sect. 3.3 provide a concise overview of the data
generation and regression methods employed, respectively. Lastly, Sect. 3.4 explains the
evaluation metrics for the regression-based heuristics.
3.1 General procedures
In this section, the general procedure of using regression methods to design heuristics for the
RCPSP is presented, and a schematic summary is shown in Fig. 1.
The ﬁgure shows that this process requires a project database that includes feasible solu-
tions (schedules) from a broad range of project instances and the corresponding data for
these instances, as described in Sect. 3.2. The database is subsequently partitioned, 80% is
allocated for training data, while the remaining 20% is reserved for test data.
The subsequent step in the process is the application of a speciﬁc regression algorithm to
the training data to generate heuristics, which will be listed and discussed in Sect. 3.3. The
reﬁnement of these regression-based heuristics is achieved through hyperparameter tuning,
which identiﬁes the optimal model parameters using the grid search technique, a widely
used approach for hyperparameter optimization in machine learning (see Goodfellow et al.,
2016 for details). This approach includes adjusting the hyperparameters across all possible
conﬁgurations to yield the best possible value. The hyperparameters we considered for each
regression model and their value are detailed in Sect. 4.1.2. Once this tuning is accomplished,
the regression models are implemented on the test data. The performance of each model is
evaluated by the makespan achieved on the test data, as reported in Sect. 3.4. Ultimately,
the best performing regression-based heuristics are selected and compared with traditional
priority rules and those designed by GPHH.

3.2 Data generation
3.2.1 Model training
The training data for constructing the regression models come from two key compo-
nents, project data and solution data. Project data serve as the independent variables
⃗x = (x1, x2, . . . , xm), capturing the features that describe the characteristics of all activ-
ities within each project instance in the training dataset, where m represents the number of
elements in ⃗x. The solution data, consisting of the start times of each activity from feasi-
ble schedules, serve as the dependent variable Y. Each activity within a project is modeled
independently, with its own feature vector and corresponding target value for the dependent
variable. The primary objective of the regression models is to learn a mapping between activ-
123

Annals of Operations Research
ity features and their corresponding start times in projects. Once trained, the regression model
predicts an estimated start time for each activity in unseen projects. This estimated start time
is not used directly as the ﬁnal schedule output; instead, it serves as a priority value in a
schedule generation scheme, which then determines the ﬁnal scheduling solution. Further
details will be discussed in the following paragraphs.
First, the activity features are determined by the attributes used in traditional priority rules,
as listed in Table 1, which form the basis of the priority functions of these rules. Consequently,
⃗x contains 23 elements, thereby setting m to 23. Given the varying range of attribute values,
normalization is performed by dividing each by the maximum value for that attribute in
the corresponding project. This normalization ensures consistency across different problem
instances.
Secondly, the known start times of activities from the projects in the training set are used
as the values for the dependent variable. The details of the dataset and the method used
to obtain these values are provided in Sect. 4.1.1. This approach makes use of a property
described in Kolisch (1996b): when start times are used as priority values for individual
activities and applied through a serial schedule-generation scheme, the resulting schedule
will be identical to the original schedule from which the start times were derived. This
property holds because, in the serial scheme, activities are scheduled sequentially in the
order dictated by their start times. Each activity is scheduled at its earliest feasible time,
ensuring that all precedence and resource constraints, which are already satisﬁed in the
original schedule, remain intact, thereby exactly reproducing the schedule. However, this
does not hold for the parallel schedule-generation scheme. Unlike the serial approach, the
parallel scheme progresses in time steps, considering all eligible activities simultaneously
at each step. Even when original start times serve as priorities, resource constraints and the
concurrent availability of multiple activities may lead to deviations from the original start
times. This means that even if an activity has an earlier start time than other activities in
the original schedule, it might not be scheduled at that same time in the parallel scheme due
to competition for resources. As a result, the ﬁnal schedule may differ from the original.
Moreover, we also need to normalize the start times before inputting them into regression
algorithms, as the start times for activities across different instances can vary signiﬁcantly.
This variation is due to various factors that can affect activity start times, including the number
of activities, activity duration, network structure, and resource tightness.
The start time values are normalized by transforming the corresponding schedule into a
standardized random key (SRK) vector, as suggested by Debels et al. (2006). This SRK vec-
tor incorporates the rank-based sequence of start times for each activity. The transformation
process consists of two steps. First, we sort the activities by their start times, placing activities
with earlier start times at the front of the sequence. Consequently, each activity i is assigned
an order Oi in this sequence. The value of Oi serves as an indicator for the activity position
withinanexecutionsequence,withlower(higher)valuesdenotingleading(trailing)positions.
In the second step, guided by literature (El-Bouri et al., 2000, 2005; Gupta et al., 2020), the
SRK value for activity i is determined, denoted as SRK i, by

The regression model is trained solely based on the mapping between activity features and

(SSGS and PSGS) are only applied in the testing phase, where the predicted values are
interpretedasactivitypriorities.Theseprioritiesarethenusedtoconstructschedules,enabling
performance evaluation based on resulting makespans. The rationale for adopting makespan
as the evaluation criterion, rather than traditional regression metrics, will be discussed in the
next section.
3.2.2 Model testing
During testing, generating data for the regression model only requires the calculation of
independent variables, i.e., the activity feature (attribute) values for each activity in the
project instances of the test dataset. These values are then input into the trained regression
model to compute predicted priority values for the activities, which are subsequently used
to construct a schedule through schedule generation schemes. No solution data are required
for the test phase.
Moreover, due to the speciﬁc nature of our problem, conventional regression metrics such
as R2 or root mean square error are not ideally suited for evaluating model performance.
This is because high predictive accuracy on start times does not necessarily lead to effective
heuristic behavior, especially considering that optimal solutions are not unique, i.e., multiple
schedules with identical makespans may have different activity start times. Therefore, we
evaluate the quality of a regression-based heuristic by applying it to the test dataset and
measuring the resulting project makespan. This approach better reﬂects the true utility of
the learned model in scheduling performance, rather than merely its predictive accuracy on
the target values. The calculation of the makespan for project instances in the test dataset is
structured into the following steps:
• Step 1: Priority value calculation for test data
– The trained model is used to calculate priority values for each activity in all projects
within the test dataset, based on the activity attribute values.
• Step 2: Makespan calculation
– The project makespan for each instance in the test dataset is determined by applying
the calculated priority values through the SSGS and PSGS algorithms.
• Step 3: Outcome determination
– The resultant makespan serves as the basis for evaluation, with the shorter makespan
from the two SGSs being selected as the best one.

3.3.1 Multiple linear regression and its three extensions
Multiple linear regression (MLR) is a commonly used statistical technique that employs
multiple independent variables ⃗x (also known as explanatory variables) to predict the outcome
of a dependent variable Y. As aforementioned, the elements (x1, x2, ..., xm) in ⃗x correspond to
the activity attributes, while Y corresponds to the (normalized) known start times of activities.
The relationship between these variables ⃗x and the dependent variable Y can be described
by:
Y = a0 + b1x1 + b2x2 + ... + bmxm
(6)
In Eq. (6), a0 is the constant term and b1, b2, ..., bm are the coefﬁcients for the independent
variables. In our application, where priority values are being determined, the constant term a0
does not inﬂuence the relative ranking of these values. Therefore, in alignment with our focus
on relative positioning, the constant term a0 will be removed in subsequent calculations.
In addition to exploring the standard version of MLR, we also assess the performance of
this regression model when implementing regularization techniques, including Ridge (Hoerl
and Kennard, 1970), Lasso (Tibshirani, 1996), and Elastic Net (Zou and Hastie, 2005). These
three regularization techniques serve as crucial additions in the ﬁeld of machine learning and
are designed speciﬁcally to enhance the performance of regression models. By constraining
the size of the regression coefﬁcients, they signiﬁcantly increase model robustness. Moreover,
these techniques do not modify the format of input and output data. Rather, they add a layer
of complexity and reﬁnement during model training, making them especially beneﬁcial when
123

Annals of Operations Research
dealing with high-dimensional data or multicollinearity issues. They are brieﬂy introduced
in the following paragraphs:
Ridge. This regularization technique, also known as Tikhonov regularization, builds upon
the ordinary least squares (OLS) estimation procedure by incorporating an L2 penalty term
into the cost function. This penalty term is equivalent to the square of the magnitude of the
coefﬁcients. In this way, the regularization encourages smaller and more stable estimates,
thereby reducing the variance, albeit at the cost of introducing a slight bias into the estimates.
By including the squares of the weights in the cost function, Ridge regression imposes
speciﬁc shrinkage on the coefﬁcients, which serves to mitigate both model complexity and
multicollinearity. However, it should be noted that while Ridge regression does shrink the
coefﬁcients towards zero, it will not make them exactly zero, and thus does not perform
variable selection.
Lasso. The least absolute shrinkage and selection operator (Lasso) technique represents
a different approach to imposing a penalty on the size of coefﬁcients. Lasso regression
introduces an L1 penalty term to the cost function, equivalent to the absolute value of the
magnitude of coefﬁcients. The key distinction between Ridge and Lasso regression lies in
the ability of Lasso to produce sparse solutions. This property of Lasso regression pushes
some coefﬁcients to become exactly zero, thus effectively performing variable selection and
providing a built-in feature selection mechanism. This characteristic is particularly beneﬁcial
for high-dimensional datasets where feature selection is important.
Elastic Net. The Elastic Net technique combines the strengths of both Ridge and Lasso
regression. It introduces a penalty term that mixes both the L1 and L2 penalties of Lasso and
Ridge in its cost function. The balance of these penalties is governed by a mixing parameter.
This combination allows Elastic Net to harness the beneﬁts of both Ridge’s solution stability
and Lasso’s sparsity. Therefore, Elastic Net can be particularly effective when dealing with
a situation where there are multiple correlated features.
3.3.2 Decision tree regression
A decision tree is a graphical model used in decision making and machine learning, where
each node represents a speciﬁc attribute, each branch corresponds to a decision rule, and
each leaf denotes an outcome, thus enabling complex relationships between inputs and out-
puts to be modeled effectively. Several learning algorithms exist for constructing decision
trees, including Iterative Dichotomiser 3 (ID3) (Quinlan, 1986), C4.5 (Quinlan, 2014), and
the Classiﬁcation and Regression Tree (CART) (Breiman, 2017). Among these, the CART
method is recognized as one of the most effective techniques (Breiman, 2017; Cherkassky
and Mulier, 2007) due to its ﬂexibility in handling both classiﬁcation and regression tasks.
Consequently, we have chosen the CART method for our research to carry out decision tree
regression.
The dependent variable Y for the decision tree regression in the normalized known start
time of each activity, and the feature vector ⃗x consists of the 23 activity attributes. In our
case, each internal node in the tree structure of CART represents a splitting criterion based
on one of the activity attributes. For example, a node might ask whether the duration of an
activity is greater than a certain threshold. Based on the answer, the data then ﬂow to one of
the two child nodes, which further reﬁnes the partition of the feature space. These partitions
eventually lead to leaf nodes, where the predicted start time Y is calculated as the mean start
time of all the activities falling within that particular region of the feature space. An example
illustrating the use of CART with the activity attributes RD, TS, and IS can be seen in Fig. 2.

The operating principle of the CART algorithm involves segmenting the entire training
database into multiple subsets based on independent variables. This segmentation process
starts with the entire database, also known as the root node, which encompasses the full train-
ing set. The tree bifurcates from the root node into left and right branches, each representing
a range of values. This bifurcation process repeats, guided by the least-square deviation
(LSD) criterion until the estimation error associated with the dependent variable reaches a
minimum. One of the distinguishing characteristics of the CART is its nonparametric nature,
which allows it to learn any form of mapping function from the provided training data (Rus-
sell and Norvig, 2010) without adhering to a particular distribution. This makes it a ﬂexible
method, albeit one that requires a larger volume of training data compared to parametric
methods.
3.3.3 K-nearest neighbor regression
The k-Nearest Neighbor (kNN) technique (Cover and Hart, 1967) is a machine learning
algorithm that utilizes historical data to ascertain the closest neighbors to a speciﬁc data point.
In this context, the term "data point" refers to an individual observation within the dataset,
commonly referred to as an "instance" in kNN terminology. In generating predictions, the
kNN algorithm considers the proximity between the new data point and a designated number
of these "instances" from the training set. This number of closest points is denoted by "k". This
algorithm can be employed to address both regression and classiﬁcation problems, similar
to the decision tree algorithm. The lazy learning characteristic of kNN contributes to its high
efﬁciency because most computation occurs while evaluating new instances (Aha, 1997).
Therefore, the need for extensive training time is eliminated. In the context of our research,
the kNN algorithm also utilizes a feature vector ⃗x consisting of the 23 activity attributes to
predict the known start time of each activity. Speciﬁcally, each data point in the training set is
a vector of these attributes, and the output or the "label" for each data point is the normalized
known start time of the corresponding activity.
When dealing with a regression problem, the kNN algorithm estimates the value of the
new data point by averaging (or giving each of them a weight) the values of the "k" closest
training data points. A variety of distance metrics can be used to calculate these closest
points. The diverse selection of metrics enhances the adaptability and applicability of the
kNN algorithm. The application of these metrics within the kNN algorithm is detailed in

3.3.4 Artiﬁcial neural network regression
An artiﬁcial neural network (ANN) is a collection of highly interconnected neurons that can
be trained to learn relationships between input and output patterns. The architecture of an

in the training set. The output layer consists of a single neuron, which is trained to predict the
start time of each activity. The architecture, i.e., the number of hidden layers and the number

input data and generate outputs. We considered a range of commonly used activation func-

3.3.5 Support vector machine regression
The support vector machine (SVM) is a supervised learning model deeply rooted in the
Vapnik-Chervonenkis dimension theory and structural risk minimization principle. It was
developed to tackle the classiﬁcation and regression tasks (Vapnik, 1999). The SVM model
implements kernel functions to conduct regression calculations. These functions have a role
of converting the lower dimensional input data into higher dimensional feature spaces. This
transformation enables the algorithm to navigate complex, non-linear scenarios by working
in a more suitable space. Similar to previous regression methods, the 23 activity attributes
constitute the elements of the input feature vector ⃗x, while the known start time of each
activity serves as the target output Y for the regression.
The core aim of the support vector machine regression (SVR) is to discover a function
that maintains the least possible deviation and exhibits the highest degree of linearity for all
training data points and their respective target outputs (Smola and Schölkopf, 2004). This
function is succinctly represented as
Y = ⃗wT ϕ(⃗x) + bsvm
(8)
In Eq. (8), ⃗w is the weight vector, capturing the learned weights for each feature in the input
vector, ϕ(⃗x) is the converting function responsible for mapping the input feature vector ⃗x into
a higher-dimensional feature space and this transformation is implicit when kernel methods
are used. bsvm denotes the bias or deviation. In this research, both linear and non-linear kernel
functions will be tested through the hyperparameter ﬁne-tuning process.
3.3.6 Genetic programming based symbolic regression
Symbolic regression is a form of regression analysis that aims to discover mathematical
expressions that best describe the relationship between independent and dependent variables.
Among various techniques, Genetic Programming (GP) is one of the most commonly used
approaches to implement symbolic regression (Augusto and Barbosa, 2000), and has been
applied in diverse prediction tasks such as estimating ﬁsh weight (Yang et al., 2021) and
shear capacity in engineering (Chaabene and Nehdi, 2021).
It should be mentioned that, in this research, GP is used purely for symbolic regres-
sion, where the goal is to generate interpretable mathematical expressions that map activity
attributes to start times. This use of GP is fundamentally different from its role in hyper-
heuristic frameworks, in which GP evolves priority rules rather than learning a direct
regression function. In our case, the generated expression directly serves as the regression
model to compute the dependent variable Y from the input features.
In this research, the functions utilized are the four arithmetic ( +, −, ×, ÷) operations, and
23 activity attributes listed in Table 1 are employed as variables. Figure 4 presents an example

A brief description of the procedures of GP is provided below (see Koza (1992) for further
details). Prior to the evolutionary process, key parameters such as tree depth and population
size are initialized. The initial population is then randomly generated by combining variables
(activity attributes) and functions (+, −, ×, ÷) into symbolic trees using the ramped half-
and-half method. This method combines fully grown and randomly grown trees across
varying depths to ensure population diversity at the start of evolution (see Koza (1992) and
also our detailed explanation in Luo et al. (2022)).
Once the initial population is created, the GP process simulates natural evolution through
three main genetic operators: selection (choosing high-quality individuals based on their
ﬁtness to reproduce), crossover (exchanging subtrees between two parent individuals), and
mutation (replacing a subtree with a newly generated one). These operators are applied
with predeﬁned probabilities. The population then evolves over successive generations, with
offspring expressions evaluated at each step. The process continues until a stopping criterion
is met (e.g., maximum number of generations or convergence threshold), after which the
best-performing individual is retained and used as the symbolic regression model.
3.3.7 Ensemble regression methods
After our exploration of individual regression algorithms, we extend our analysis to consider
ensemble regression methods. Ensemble methods are an approach that uses multiple learning
algorithms/models to obtain superior predictive performance than could be achieved from
any of the individual learning algorithms/models alone (Opitz and Maclin, 1999). These
ensemble methods are based on single regression models, eliminating the need to modify the
input-output data formats. This makes them highly convenient for extending the capabilities
of our top-performing models, while also serving as a comprehensive framework designed
to maximize predictive performance by effectively managing the strengths and weaknesses
of each base model.
Following our evaluation of each previously mentioned regression algorithm, we selected
the top three models based on their performance. To further improve their performance,
we applied ensemble methods to each of these selected models. Speciﬁcally, we utilized
bagging (Efron, 1992) and boosting (Schapire, 1990), both of which are among the most
widely adopted techniques in the ﬁeld. A brief overview of these two ensemble methods is
presented below:
Bagging. Bagging (acronym for bootstrap aggregating) is among the earliest and most
widely used ensemble methods. The principal idea of bagging involves the construction of
independent predictors using samples from the training set, and then averaging (or voting) the
123

Annals of Operations Research
output of these predictors. Variety in bagging is achieved by utilizing bootstrapped replicas of
the original data. Different training sub-datasets are randomly drawn, with replacement, from
the training dataset. From these subsets, separate models are produced and used to predict the
entire data. The estimates from the various models are then aggregated using the mean for
regression problems or majority voting for classiﬁcation problems. A more comprehensive
discussion on bagging is available in Breiman (1996), where it is demonstrated that bagging
ensembles of decision trees and regression trees can outperform single trees.
Boosting. The boosting tech deployed in this research is the adaptive boosting technique
(AdaBoost, Freund and Schapire (1996)). The characteristic of AdaBoost is that it uses the
initial training data to generate a weak learner (i.e., the single regression model), then adjusts
the distribution of the training data according to the prediction performance for the subsequent
weak learner training. It should be noted that the training samples with lower prediction
accuracy in the previous step are given more attention in the following step. Eventually,
the weak learners are integrated into a stronger learner, with each being assigned different
weights.
3.4 Model evaluation
After calculating the makespan for each instance in the test set, we use the following metrics
to quantify performance across all test project instances:
• SumMakespan: The summation of the obtained makespan.
• AvgDevCPM: The average deviation (in %) from the critical path length.
• AvgDevLB: The average deviation (in %) from the best known lower bound (reported in
Vanhoucke and Coelho (2018)).
• AvgDevBKS: The average deviation (in %) from the best known solution in cases where
no lower bound data are available.
Moreover, the performance of regression-based heuristics is evaluated relative to seven
distinct points of reference:
• GP1: The best-performing GP-designed rule reported in Luo et al. (2022).
• WCS: The well-known worst case slack rule, originally proposed by Kolisch (1996a).
The WCS rule can only be implemented with PSGS. For each eligible activity, it computes
the remaining slack in the worst case if the activity is not selected in the current scheduling
iteration.
• LFT-P: The latest ﬁnish time rule, a well-known classical benchmark priority rule, imple-
mented with PSGS.
• LFT: The LFT rule implemented with both SGS variants, SSGS and PSGS. The better
of the two solutions is selected as the ﬁnal result.
• SinglePR: The best-performing traditional rule among the 36 priority rules examined
in Luo et al. (2022). Speciﬁcally, for each dataset, AvgDevLB is calculated for all 36
priority rules, and the rule achieving the lowest AvgDevLB is selected as the SinglePR
for that dataset.
• AllPR: The best possible performance obtained by using all 36 priority rules as described
in Luo et al. (2022). In particular, this method follows the multi priority rule approach,
also known as the "Multi Pass Methods" in the literature (see Kolisch et al. (1999) and
Hartmann and Kolisch (2000)). In this approach, each instance is solved independently
with all priority rules, and the solution with the best makespan is selected as the ﬁnal
outcome.Thisapproachhasmeritforsmall-scaleinstances(furtherdiscussionisprovided
in Appendix B).
123

Annals of Operations Research
• Dominance: The percentage of instances in which other methods achieve equal or better
results compared to SinglePR.
For the algorithmic baselines that do not explicitly specify the SGS variant (GP1, SinglePR,
and AllPR), we evaluate each rule under both the SSGS and PSGS and retain the better result
for comparison, as this allows each rule to demonstrate its full potential while requiring only
marginal additional computational effort (see the experiments in Sect. 4 and Appendix B for
details).
In addition, to evaluate the statistical signiﬁcance of the observed performance differ-
ences, we conduct the Wilcoxon signed-rank test with a signiﬁcance level of 0.05. This
non-parametric test allows us to conﬁrm whether improvements achieved by regression-
based heuristics over baseline methods are statistically meaningful across different datasets.

4.2.1 The performance of single regression methods
This section presents the results of heuristics based on single regression models on the test
set. In this experiment, we compare the performance of regression-based heuristics on the
test set with traditional rules and GP-designed rule (GP1).
To assess whether the performance improvements of regression-based heuristics over
baseline methods are statistically signiﬁcant, we employ the Wilcoxon signed-rank test
with a signiﬁcance level of 0.05, as brieﬂy mentioned in the model evaluation section.
Speciﬁcally, for each test instance (with the rationale for this implementation discussed
in Luo et al. (2023)), we compute the performance difference between the best regression-
based heuristic and each baseline (SinglePR, AllPR, or GP1) as:
Difference = resultregression −resultbaseline.
(9)
The null hypothesis H0 posits that the median of the differences is zero. The outcomes of the
statistical test are noted in the performance tables using the symbols "(+)", "(−)", and "(=)".
Speciﬁcally:
• "(+)" indicates that the regression-based heuristic performs signiﬁcantly better than the
baseline (median of differences < 0),
• "(−)" indicates signiﬁcantly worse performance (median of differences > 0),
• "(=)" denotes that no statistically signiﬁcant difference is observed.
The detailed comparison results are presented in Table 5, which includes performance
metrics, statistical test outcomes, and the total CPU time (in seconds) required to process
each subset of the PSPLIB.
In Table 5, the best-performing regression-based heuristic for each dataset is highlighted
in bold. Based on these results, we observe that Lasso, Elastic Net, and ANN consistently
perform well across different subsets. Notably, their CPU time is similar to that of base-
line methods, indicating comparable computational efﬁciency. In contrast, AllPR requires
approximately thirty times the runtime of a single priority rule, due to its exhaustive eval-
uation process. All these three regression-based heuristics exhibit a signiﬁcant dominance
percentage, with up to 96.88% of instances performing better than or at least equal to Sin-
glePR, as further validated by the statistical test results. Consequently, these three models
are chosen for further improvement using ensemble methods. Notably, the Lasso regression
model stands out with the highest performance among all the regression-based heuristics. It
outperforms SinglePR across almost all subsets of the PSPLIB and even outperforms GP1
on the J30 dataset.

4.2.2 The performance of ensemble regression methods
After the top three performing models are determined, each model is individually enhanced
using either the bagging or boosting ensemble method, resulting in six different combinations
of enhancements. Since ensemble methods also require a hyperparameter ﬁne-tuning, and
the best performing setting may vary for different combinations, therefore, each combination
will also undergo a hyperparameter ﬁne-tuning process. The considered hyperparameters
for each ensemble method are shown in Table 6, and the performance of each combination
after the tuning process is listed in Table 7. The full search ranges used during the tuning
process are provided in Appendix C. An asterisk (*) in Table 7 marks cases where applying
the ensemble method to a given single regression model resulted in a drop in performance on
a speciﬁc dataset. The data corresponding to the best-performing single model within each
ensemble type (i.e., bagging or boosting) are highlighted in bold.
The results presented in Table 7 highlight two noteworthy ﬁndings. Firstly, the perfor-
manceofthethreeselectedsingleregressionmodelscanbesigniﬁcantlyimprovedbyutilizing
ensemble methods in most cases. For completeness, we also conducted experiments applying
ensemble methods to non-top-performing regression models. Although these combinations
show some performance improvements, they still lag behind the selected top-performing
models. Therefore, their detailed results are provided separately in Appendix D for refer-
ence, and are not further discussed here. Secondly, the Lasso and Elastic Net regression
methods, when employed with ensemble techniques, demonstrate a high level of competi-
tiveness compared to traditional rules or GP1, as conﬁrmed by the results of statistical tests.
The table reveals that the Lasso regression with the bagging technique yields the best results.
Additionally, the Elastic Net regression with the boosting approach also achieves compara-
ble performance. These two combinations outperform SinglePR and GP1 on the J30 and J60
datasets. On the J90 and J120 datasets, although they do not surpass the performance of GP1,
they still demonstrate superior performance compared to SinglePR.
In summary, the results of this experiment show both the efﬁcacy of ensemble methods
in improving regression model performance and the effectiveness of utilizing regression
models for designing heuristics for the RCPSP. The Lasso regression with bagging and Elastic
Net regression with boosting emerge as particularly competitive combinations, showcasing
superior performance across various datasets. Although they may not always surpass the
performance of GP1, they outperform traditional rules and establish themselves as strong
alternatives. However, the performance of ANN model with ensemble methods is inferior
to these two powerful combinations and even SinglePR, given its inferior performance, the
ANN model will be discarded in our subsequent experiments. Instead, we will employ Lasso
and Elastic Net regressions, each coupled with both bagging and boosting techniques, as the
representatives for our regression-based heuristics method.

In this section, we conducted performance tests on supplementary datasets independent of

These datasets encompass projects with a wide range of project-level indicators. The results
related to the benchmark datasets are presented in Sect. 4.3.1, while the outcomes for the
newly generated large projects are detailed in Sect. 4.3.2. Finally, the performance on the

4.3.1 The performance on benchmark datasets
In this experiment, the performance of the regression-based heuristics on the benchmark
datasets is presented. Comprehensive results are provided in Table 9. The best-performing
regression-based heuristic for each dataset is highlighted in bold. Moreover, to equitably
compare the generalization capabilities of GP-designed rules and regression-based heuris-
tics, we introduce a new rule "GP_J30," which is designed using the GPHH method and
trained on a subset of 96 projects extracted from the J30 dataset. This is consistent with
the convention of designing priority rules using the GPHH method. Using the entire dataset
would result in a long training time without improving performance. (Details can be found
in Luo et al. (2022).)
Table 9 evidences the superior performance of regression-based heuristics over SinglePR,
with speciﬁc note that Lasso regression coupled with bagging and Elastic Net regression
enhanced by boosting consistently outperformed SinglePR across all datasets, underscoring
their robust generalization capabilities. However, in this experiment, the regression-based
heuristics could not outperform GP1. This outcome can be partly attributed to the fact that
GP1 was designed using a training set that included both the J30 dataset and the RG30 dataset
(see Luo et al. (2022)). It is important to note that the RG30 dataset was generated by the
RanGen generator, which our regression models were never exposed to during the training
process. As a result, GP1 has a natural advantage in solving project instances generated by
RanGen, making the comparison less equitable. This observation gains further support from
the performance of GP_J30, which performs similarly to our regression-based methods but
lags behind GP1. This implies that the superior performance of GP1 can be attributed to its
diverse training data rather than the efﬁcacy of the genetic programming approach itself.
Furthermore, when considering the "Dominance" metric, the regression-based heuristics
consistently outperformed the best GP-designed rule in all test sets, except for the RG300
dataset. Thus, in the majority of cases, utilizing regression-based heuristics is likely to result
in improved outcomes.

4.3.2 The performance on large projects
In this experiment, we present and compare the performance of regression-based heuristics
on large project datasets with traditional priority rules and the GP-designed rule. Moreover,
given that one of the advantages of the priority rules approach is their ability to rapidly
solve large projects while maintaining commendable solution quality, we have included the
results of the genetic algorithm (GA) proposed in Debels and Vanhoucke (2007) truncated
after computing 1,000, 5,000, and 50,000 schedules (marked as DV_1000, DV_5000, and
DV_50000, respectively). To provide a full picture of the experimental setup, we also include
the CPU time of the GA, though it warrants clariﬁcation. These values were obtained using a
different programming platform on a high-performance computing (HPC) system equipped
with several thousand cores and parallel computing capabilities. As such, the reported runtime
is not directly comparable to those of other methods, which were executed on a standard
desktop environment, and should be interpreted as a reference value only. Given the size of
the problem instances and the extensive number of schedules that must be evaluated, it is
reasonable to conclude that executing the same GA implementation on a personal computer
would result in signiﬁcantly higher computational time. The detailed results are displayed in
Table 10. The best-performing regression-based heuristic for each dataset is highlighted in
bold.
Several interesting conclusions can be drawn by analyzing the data in Table 10. Firstly,
regression-based heuristics demonstrate superior generalization capabilities by outperform-
ing SinglePR across all three tested datasets. This is particularly signiﬁcant compared to GP,
which outperforms SinglePR in the Multi-projects and Large-projects/P datasets but not in
Large-projects/SP. The ability of regression-based heuristics to effectively apply knowledge
from small to large projects is a notable advantage.
Secondly, across all three datasets evaluated in this experiment, the regression-based
heuristics demonstrate performance comparable to that of AllPR, indicating an edge over
traditional rules. Moreover, the contrast with previous comparisons involving smaller projects
is noteworthy: AllPR demonstrated a clear advantage in solving those small project instances,
but this advantage is less pronounced in large projects and sometimes even inferior to GP1 and
regression-based heuristics. Therefore, the efﬁciency of deploying newly designed single-
priority rules (that outperform SinglePR) on large projects becomes apparent. Compared to
AllPR, they require signiﬁcantly fewer schedules to be computed (tens of times fewer) while
still achieving better or comparable results, as validated by the statistical test results. This
observation highlights the practical value of designing new priority rules: in large projects,
the primary computational burden stems from applying schedule generation schemes, and
executing multiple traditional rules (as in AllPR) becomes increasingly expensive without
guaranteeing superior performance due to the vastness of the solution space. In contrast, a
single well-designed priority rule, such as those derived through regression or other machine
learning algorithms, can signiﬁcantly reduce scheduling overhead while offering the potential
to outperform traditional rule combinations. These ﬁndings further reinforce the beneﬁts of
regression-based or other machine-learning-based heuristics for solving large projects.
Finally, our regression-based heuristics demonstrate remarkable competitiveness in large
parallel projects compared to a genetic algorithm. They outperform DV_1000 and DV_5000
in the Multi-projects and Large projects/P datasets, respectively. This is further conﬁrmed
by Table 11, which provides a direct head-to-head comparison by listing the percentages of

4.3.3 The performance on empirical project data
The best-performing regression-based heuristic methods from previous experiments are
applied to solve the empirical project instances. Table 12 provides detailed information on
both the project instances used for testing and the results of this experiment, with makespan
used as the performance metric.
The results for these 10 instances reveal the following insights: First, the solution space
remained smaller than we had anticipated, even after carefully selecting and combining the
source projects into larger instances for testing. Consequently, the performance differences
across methods were not as signiﬁcant as we expected. However, in 3 of the 10 instances
(highlighted in bold), regression-based heuristics outperformed the best single traditional

artiﬁcial data on which they were trained. As a result, these methods also prove competitive
when applied to other real-world projects.
5 Conclusion
In this research, the regression models were employed to design heuristics for the RCPSP.
For that purpose, we ﬁrst investigated the performance of nine commonly used regression
models, and our experiments have shown that Lasso regression, Elastic Net regression, and
ANN regression are the best three among them. These models were subsequently selected
to be enhanced by an ensemble method. After examining the performance of the enhanced
regression models, we found that the combination of Lasso regression enhanced by the
bagging method excelled all the traditional rules on all test sets. Although this approach
outperformed the GP-designed rule in two test sets, particularly those containing smaller-
scale instances, it was less effective on two other test sets with larger instances. Nonetheless,
the overall performance and the promising results in the smaller-scale projects highlight
the value of the regression-based heuristics, positioning it as a noteworthy alternative to
GP-designed rules.

To test the generalization capability of the regression-based heuristics, we evaluated their
performance using three well-known benchmark datasets from the literature, along with three
newly generated large project datasets. These datasets are independent of our primary data.
Our regression-based heuristics outperformed traditional rules on the benchmark datasets
from the literature, although they were not always as effective as the GP-designed rule.
Nonetheless, when the training datasets share similar characteristics, the performance of
GP-designed rules closely aligns with that of our regression-based heuristics. Moreover,
regression-based heuristics surpassed all the traditional priority rules on large project datasets.
It even exceeded the genetic algorithm after the latter had computed thousands of schedules.
In addition, regression-based heuristics also shown strong competitiveness on the empirical
project datasets. Therefore, our regression-based heuristics exhibit promising generalization
potential.
According to the results obtained, we conclude that using supervised learning approaches,
especially regression algorithms, can be used to design heuristics that outperform traditional
priority rules for the RCPSP in most cases. Moreover, these newly generated heuristics have
their advantage over the GP-designed rules, such as superior performance on small projects of
the test data (which has similar characteristics as the training data) and better generalization
capability on large projects. Given that regression-based heuristics offer other advantages,
such as signiﬁcantly lower training runtime (requiring only several minutes compared to
the hours or even days needed by GPHH), we believe that using regression algorithms is
a promising approach for designing new heuristics for the RCPSP and deserves increased
attention from researchers.
However, it is also important to acknowledge that these improvements come at the cost
of training overhead, and the observed performance gains are signiﬁcant only on selected
subsets. Therefore, more research is needed to further improve their efﬁciency and robustness
across a wider range of instances.
The future work of our study primarily encompasses four distinct aspects. First, there
is an avenue to explore more comprehensive and advanced prediction models that could
enhance the performance of the regression-based heuristics, making this an appropriate and
valuable direction for further research. Second, the innovative method of utilizing regres-
sion algorithms to design heuristics could be broadened to tackle other project scheduling
problems, including those with different objective functions such as resource leveling or net
present value. Third, this regression-based approach can be adapted to solve extended vari-
ations of the RCPSP, including its dynamic version and the multi-project problem. Finally,
investigating more appropriate data patterns would enable the regression models to learn the
relationships between dependent and independent variables more easily and effectively. This
exploration could provide new insights and further the efﬁcacy of our proposed methods.
Appendix A Pseudocode for deploying SGSs with a priority list
This appendix presents the pseudocode illustrating how a priority rule is applied to construct
a priority list, and how the resulting list is subsequently used by the schedule generation
schemes to generate schedules.
Algorithm 1: Construction of the priority list As described in Sect. 2.2, a priority list is a
permutation of project activities in which each activity appears only after all its immediate
predecessors. Let g denote the iteration step in which one activity is selected and added
to the priority list. At step g, Eg represents the set of eligible activities (i.e., those whose
123

Annals of Operations Research
predecessors have all been scheduled). The priority value of activity i is denoted by pvi, and
its position in the priority list is odi. The set of immediate predecessors of i is represented by
I Pi. We assume that a lower priority value corresponds to a higher priority. The procedure
for generating the priority list PL is outlined below.
Algorithm 1: Construction of the priority list
Initialization: PL ←[0], odi ←0, ∀i ∈A
for g ←1 to n + 1 do
Eg ←{ i ∈A \ PL | I Pi ⊆PL };
i∗←arg mini∈Eg pvi;
odi∗←g;
Append i∗to PL;
end
return PL
Algorithm 2: Applying SSGS to generate a schedule from a priority list Let si represent the
start time assigned to activity i in the resulting schedule. The remaining capacity of resource
k at time t is denoted by ˜Rk(t), and ESi refers to the earliest start time of i that satisﬁes
all precedence constraints. The following algorithm outlines how the SSGS operates when
given a priority list that respects activity precedence.
Algorithm 2: Applying SSGS to generate a schedule from a priority list
Initialization: s0 ←0,
˜Rk(0) ←Rk, ∀k ∈R
for g ←1 to n + 1 do
i ←{ j ∈PL | od j = g };
ESi ←max j∈I Pi (s j + d j);
si ←min{ t | t ⩾ESi, ri,k ⩽˜Rk(τ), ∀k ∈R, ∀τ ∈[t, t + di) };
Update:
˜Rk(τ) ←˜Rk(τ) −ri,k, ∀k ∈R, ∀τ ∈[si, si + di);
end
return sn+1
Algorithm 3: Applying PSGS to generate a schedule from a priority list The PSGS algorithm
proceeds by incrementing time. At each iteration stage g, a current scheduling time tg is
considered. Activities scheduled up to this point are either part of the active set Active(tg)
(i.e., activities still in execution at time tg) or belong to the completed set Cg. The decision
set Decg includes all activities that can be initiated at tg without violating precedence and
resource constraints. The ﬁnish time of activity i is denoted by fi. The algorithm for applying
PSGS with a priority list is presented below:

Algorithm 3: Applying PSGS to generate a schedule from a priority list
Initialization: g = 0, s0 = 0, tg = 0, Active(0) = {0}, C0 = {0}, ˜Rk(0) = Rk

g = g + 1;

Appendix B Evaluation of schedule generation schemes and priority

B.1 Evaluation of schedule generation schemes
This appendix provides additional evidence supporting the use of both schedule generation
schemes (SSGS and PSGS) when evaluating priority rule heuristics. To illustrate the effect
of SGS choice on performance, we examine three representative traditional priority rules
(LFT, LST, and MTS). Each of which has been the best-performing single priority rule (i.e.,
SinglePR) for at least one subset in our experiments.
We use the PSPLIB dataset for this analysis because it is the most widely used dataset in
the literature for algorithm evaluation in project scheduling. In addition, PSPLIB provides
instance sets with gradually increasing problem sizes (J30, J60, J90, J120), rather than making
a sudden jump in scale (for example, from RG30 to RG300). This progressive structure allows
a clear and systematic examination of SGS performance as project size increases. Moreover,
each size class forms an independent subset, which prevents results from being affected by
mixing instances of different scales.
For each rule, we report two types of results across each subset of the PSPLIB:
• Overall performance, including the sum of makespan and statistical tests comparing
SSGS, PSGS, and Both (best of the two). Statistical signiﬁcance is also assessed using
the Wilcoxon signed-rank test (signiﬁcance level 0.05), where "(+)", "(−)", and "(=)"
indicate that the left-hand method performs signiﬁcantly better, signiﬁcantly worse, or
shows no signiﬁcant difference, respectively.
• Head-to-head instance-level comparison, showing the percentage of instances where one
SGS variant performs better, equally well, or worse than the other.
The results are shown in tables 13, 14, 15, 16, 17 and 18.
The results show that PSGS generally outperforms SSGS across most instance classes.
However, using both SGS variants yields further performance gains beyond using PSGS
alone. Across the three representative rules, the Both setting achieves statistically signiﬁcant
123

Annals of Operations Research
improvements in all cases. The head-to-head comparisons also indicate that SSGS still ﬁnds
better solutions than PSGS on a non-trivial fraction of instances. Importantly, the CPU time
results reported in Sect. 4 show that adding SSGS on top of PSGS incurs only marginal
overhead while improving solution quality.
Therefore, using both SGS variants preserves the advantages of PSGS, leverages instances
where SSGS performs better, and incurs negligible computational cost. Based on this empir-
ical evidence, we decided to use the Both setting in all experiments.
B.2 Evaluation of priority rule combinations
To assess the effectiveness of selecting the best solution from multiple priority rules, we
compare the performance of AllPR and SinglePR across all datasets (empirical project data
excluded). The result is shown in Table 19.
Table 19 shows that AllPR consistently outperforms SinglePR across all datasets, with
statistically signiﬁcant improvements. This conﬁrms that combining multiple priority rules
can lead to better performance. However, as shown earlier in the experiments of Sect. 4,
AllPR requires more than thirty times the CPU time of SinglePR on average. While this
overhead may be acceptable for small instances, it becomes impractical for large projects.
This motivates our approach: rather than exhaustively evaluating traditional rules, we develop
learning-based heuristics that achieve competitive or superior performance at far lower com-
putational cost.

AppendixDPerformanceofensemblemethodswithnon-top-performing
single regression models
This appendix reports additional experiments in which ensemble methods (bagging and
boosting) were applied to regression models that were not among the top three performers
in the initial evaluation. These models include Linear, Ridge, Decision Tree, kNN, and
Symbolic regression. The aim of this supplementary experiment is to verify whether ensemble
techniques can signiﬁcantly enhance the performance of weaker base models.
For each of these models, both bagging and boosting were applied, and a hyperparameter
tuning process was conducted following the same procedure described in Sect. 4.2.2. The
tested hyperparameter ranges were identical to those used for the top-performing models and
are summarized in Appendix C.
The results of these experiments are presented in Tables 21 and 22. Table 21 reports the
best SumMakespan on the validation set for each ensemble-single model combination after
hyperparameter tuning. Table 22 summarizes the corresponding performance on the test set,
which corresponds to the PSPLIB described in Sect. 4.1.1.
Overall,whileensemblelearningprovidessomeperformanceimprovementsforthesenon-
top-performing models, the gains are generally limited. In all cases, the ensemble-enhanced
versions still perform worse than the SinglePR baseline. Given their relatively poor perfor-
mance on the test set, these combinations were not further evaluated on the supplementary
datasets. These ﬁndings reinforce our earlier decision to focus on Lasso and Elastic Net as the
most promising base learners in the context of ensemble regression for heuristic designing
in the RCPSP.
It is worth noting that the SVR model was excluded from these ensemble-single model
experiments due to its prohibitively long training time. As shown in Table 8, SVR requires
over 1,000 seconds to train even as a single model, whereas most other models ﬁnish within
approximately 20 seconds. Incorporating SVR into ensemble frameworks, particularly boost-
ing, would substantially increase computational overhead. Due to both its high time cost and
limited standalone performance, SVR was excluded from this experiment.
